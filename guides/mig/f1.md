# Phase 1: Migrate to Producer–Worker Architecture (ONE job)

### 🎯 Goal of Phase 1

- Keep **mail sending working exactly as today**
- Introduce **clear boundaries**
- Make adding future jobs a **zero-refactor operation**

No new infra. No new features.

---

## Step 0 — Freeze behavior (5 minutes)

**Purpose:** You must know what “working” means before refactoring.

- Identify:
  - where mail jobs are enqueued
  - where worker processes them

- Trigger 1 mail manually (or via panel)
- Confirm it works

👉 This is your reference point.

**DO NOT change code yet.**

---

## Step 1 — Rename infra package (semantic fix)

### Why

Right now `queue` sounds like _mail-specific logic_.
We want it to mean _job execution infra_.

### Action

Rename:

```
packages/queue → packages/job-runtime
```

- Update import paths
- Do **not** change exports or logic

### Result

- Code works exactly the same
- Semantics are now correct

📌 Commit:

```
chore: rename queue package to job-runtime
```

---

## Step 2 — Create `jobs/` folder (no wiring yet)

### Why

We want to separate:

- **what the job is**
- from **how it runs**

### Action

Create:

```
packages/jobs/
```

Do nothing else yet.

📌 Commit:

```
chore: add jobs package
```

---

## Step 3 — Extract the mail job definition

### Why

Mail logic should not live inside worker or queue infra.

### Action

Create:

```
packages/jobs/send-mail.ts
```

Move **only**:

- job name string
- payload type
- mail handler logic

Out of:

- worker code
- queue code
- web code (if embedded)

This file should answer:

> “What does ‘send mail’ mean?”

No BullMQ imports here.

📌 Commit:

```
refactor: extract send-mail job definition
```

---

## Step 4 — Introduce typed enqueue helper

### Why

This is where TypeScript _actually_ protects you.

### Action

Inside `packages/job-runtime`, create:

```
enqueueSendMail.ts
```

This file:

- imports job name + payload type
- wraps `queue.add(...)`

Then:

- replace **all direct** `queue.add("mail", …)` calls
- with `enqueueSendMail(...)`

### Result

- producers are type-safe
- job name typos are impossible
- future jobs won’t touch existing code

📌 Commit:

```
refactor: add typed enqueue helper for send-mail
```

---

## Step 5 — Simplify worker to job-based dispatch (still ONE job)

### Why

Worker should not be “mail-specific”.
It should be “job-executor”.

### Action

In `apps/worker`:

- import `SEND_MAIL_JOB` and `sendMailHandler`
- dispatch based on `job.name`

Since you still have one job, this will be:

- one `if`
- or one `switch`

No registry yet. No abstraction explosion.

📌 Commit:

```
refactor: worker dispatches by job name
```

---

## Step 6 — Stop. Validate. Do NOT add more.

At this point:

- mail still works
- architecture is clean
- future jobs fit without refactor

Do:

- run worker
- enqueue mail
- confirm delivery

If this passes → Phase 1 is DONE.

---

# What you explicitly do NOT do in Phase 1 ❌

- ❌ add S3
- ❌ add second job
- ❌ add multiple queues
- ❌ add job registry
- ❌ optimize concurrency
- ❌ over-type the worker

Those are **Phase 2+ problems**.

---

# Phase 1 Success Criteria (checklist)

You should be able to say YES to all:

- Can I add a new job without touching mail code? ✅
- Is mail logic isolated from infra? ✅
- Is queue usage hidden behind helpers? ✅
- Did behavior remain unchanged? ✅

If yes → architecture migration succeeded.
