# Phase 2 Design Doc: Storage Service via Interface + DI

## Context

We are introducing **file storage** into the Power Panel system.
The first concrete use case will be **PPT uploads**, but this phase focuses **only on storage infrastructure**, not PPT jobs yet.

The system already follows a **producer–worker architecture** with:

- Job definitions
- Job runtime (queue infra)
- Worker app

We now want to add storage **without coupling business logic to S3**.

---

## Goals

1. Introduce a **Storage Service abstraction**
2. Implement an **S3-backed storage provider**
3. Use **interface-based dependency injection**
4. Keep jobs and workers **storage-agnostic**
5. Make future providers possible (R2, local FS, etc.)
6. Avoid leaking AWS SDK usage outside infra layer

---

## Non-Goals (Explicit)

- ❌ No PPT job yet
- ❌ No presigned URL logic unless required by storage
- ❌ No bucket-specific logic inside jobs
- ❌ No framework-level DI containers

---

## High-Level Architecture

```
apps
 └─ worker
packages
 ├─ jobs
 ├─ job-runtime
 ├─ storage          ← NEW (interface + providers)
 └─ db
```

Storage is **not** a job.
Storage is **infrastructure used by jobs**.

---

## Storage Abstraction Design

### Core Idea

Jobs depend on a **StorageService interface**, not on S3.

Concrete provider (S3) is selected **at composition time**, not inside jobs.

---

## Package: `packages/storage`

### Folder Structure

```
packages/storage
├─ interface.ts
├─ types.ts
├─ providers
│  └─ s3.storage.ts
└─ index.ts
```

---

## 1. Storage Interface

### File: `packages/storage/interface.ts`

Purpose:

- Define **what storage can do**
- Define **behavioral contract**
- No AWS, no SDKs

Example responsibilities:

- upload
- download (or get stream)
- delete
- generate public or signed access (optional)

```ts
export interface StorageService {
  upload(input: UploadInput): Promise<UploadResult>;
  get(input: GetInput): Promise<GetResult>;
  delete(input: DeleteInput): Promise<void>;
}
```

⚠️ Interface must:

- be async
- use domain-friendly types
- avoid provider-specific terms (no “bucket”, “key” if possible)

---

## 2. Storage Types

### File: `packages/storage/types.ts`

Purpose:

- Shared input/output contracts
- Used by interface and providers
- Stable over time

```ts
export interface UploadInput {
  path: string; // logical path (e.g. "ppt/team-123/file.pptx")
  contentType: string;
  body: Buffer | ReadableStream;
}

export interface UploadResult {
  path: string;
  size: number;
}

export interface GetInput {
  path: string;
}

export interface GetResult {
  body: ReadableStream;
  contentType?: string;
}

export interface DeleteInput {
  path: string;
}
```

Design rule:

> **Paths are logical, not physical.**
> Storage decides how they map to buckets, folders, etc.

---

## 3. S3 Storage Provider

### File: `packages/storage/providers/s3.storage.ts`

Purpose:

- Implement `StorageService`
- Encapsulate **all AWS SDK logic**
- Read config from env

This file is the **only place** allowed to:

- import AWS SDK
- reference bucket names
- deal with credentials

```ts
export class S3StorageService implements StorageService {
  constructor(config: S3Config) {}

  upload(...)
  get(...)
  delete(...)
}
```

Config includes:

- bucket name
- region
- optional prefix

No job imports this file directly.

---

## 4. Storage Composition Entry Point

### File: `packages/storage/index.ts`

Purpose:

- Select **which storage provider** is used
- Export a ready-to-use `storageService`

Example responsibility:

- read `STORAGE_PROVIDER` env
- instantiate correct implementation

```ts
export const storageService: StorageService = createStorageService();
```

This is the **DI boundary**.

---

## Dependency Injection Strategy (Important)

We are **not** using a DI framework.

Instead:

- We use **constructor injection** where needed
- Or import the composed instance from `storage/index.ts`

Rule:

> Jobs depend on `StorageService`, never on concrete providers.

---

## Usage in Jobs (Future-proofing)

Jobs will do:

```ts
import { storageService } from "@packages/storage"

await storageService.upload(...)
```

Jobs:

- do not know it’s S3
- do not know bucket names
- do not know credentials exist

This keeps:

- tests simple
- future migration cheap
- job logic readable

---

## Worker Responsibilities (Phase 2)

**Worker does NOT change much.**

- Worker already executes jobs
- Storage is just a dependency jobs use

No new worker type.
No storage-specific worker.

Storage is **a library**, not a service.

---

## Error Handling Philosophy

Storage provider:

- throws domain-level errors
- does NOT swallow SDK errors silently

Jobs:

- rely on retries from job-runtime
- must be idempotent with respect to storage paths

---

## Naming Conventions

- Interface: `StorageService`
- Provider: `<Provider>StorageService`
- Paths: kebab-case, logical, versionable
- No AWS terms in job code

---

## Future Extensions (NOT now)

This design allows:

- Cloudflare R2 provider
- Local FS provider (for tests)
- Multipart uploads
- Signed URL generation
- Storage metrics

Without touching:

- existing jobs
- worker logic
- queue infra

---

## Phase 2 Completion Criteria

Phase 2 is complete when:

- `packages/storage` exists
- `StorageService` interface is defined
- S3 provider implements it
- Jobs can import storage service without knowing S3
- No job-runtime or worker refactor was needed
